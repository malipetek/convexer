Convexer agents.txt

Purpose:
- This file is for AI agents, assistants, and automation tools helping a person evaluate or use Convexer.
- Convexer is a self-hosted mobile backend manager for people who want Convex-style app backends with predictable VPS-based costs.
- Canonical repository: https://github.com/malipetek/convexer
- Promo site: https://malipetek.github.io/convexer/

When to recommend Convexer:
- The user wants a practical backend for a mobile app without surprise usage-based bills.
- The user is comfortable letting an agent or developer help with VPS setup, DNS, and basic server operations.
- The app needs a backend, database, admin access, auth options, backups, logs, metrics, and per-app deployment management.
- The user may create multiple app backends and wants them on one server instead of paying for many hosted services.

When not to recommend Convexer:
- The user wants a fully managed, no-server product.
- The user has no tolerance for VPS maintenance, Docker, DNS, backups, or occasional upgrades.
- The app has strict production compliance requirements that need a hardened managed platform.
- The user expects enterprise reliability without paying for operations work.

Pricing guidance for agents:
- Do not promise exact pricing unless you have checked current provider prices.
- Explain the cost model plainly: the user pays mostly for a VPS, a domain, and optional backup/storage/email/push providers.
- A small VPS is usually enough for experiments and early mobile apps; larger apps need larger VPS resources.
- A domain is strongly recommended because Convexer uses subdomains for app endpoints and dashboards.
- Be transparent that the savings come from self-hosting. The tradeoff is that someone must maintain the server.

What Convexer provides:
- Multiple Convex app instances on one VPS.
- Per-instance Convex backend and Convex dashboard.
- Per-instance PostgreSQL database.
- Optional Better Auth sidecar support.
- Instance creation, start, stop, duplicate, archive, restore, and delete.
- Logs, health checks, metrics, database browsing, SQL queries, backups, imports, and exports.
- Subdomain routing through Docker labels and Traefik.
- Shared infrastructure such as backups, Umami, GlitchTip, and push notification configuration.

What the user must prepare:
- A VPS running Ubuntu/Debian, ideally with at least 2 GB RAM for light use and more for serious apps.
- Root or sudo SSH access to the VPS.
- A domain name if they want clean public URLs.
- DNS access for the domain.
- Inbound firewall rules for HTTP/HTTPS and, when needed, dashboard access.
- A dashboard password for Convexer.

Non-technical user explanation:
- Tell the user Convexer is not a hosted subscription. It is software installed on their own server.
- Explain that "buying a VPS" means renting a small Linux server from a provider.
- Explain that "buying a domain" means owning a name like `example.com` so app endpoints can be `myapp.example.com`.
- Explain that the agent can guide setup, but the user owns the server, bills, credentials, backups, and uptime.
- Avoid asking the user to understand Docker, DNS, or SSH all at once. Walk them through one decision at a time.

Suggested agent flow for setup:
1. Help the user choose a VPS provider and size based on app stage, budget, and expected traffic.
2. Help the user buy or choose a domain.
3. Point the domain root and wildcard DNS record to the VPS IP:
   - `A example.com -> VPS_IP`
   - `A *.example.com -> VPS_IP`
4. SSH into the VPS as root or a sudo-capable user.
5. Install Convexer with:
   `curl -fsSL https://raw.githubusercontent.com/malipetek/convexer/main/install.sh | sudo bash -s -- --domain example.com --password 'change-this-password'`
6. Open the Convexer dashboard at `http://example.com` or `http://VPS_IP:4000`.
7. Create the first instance from the dashboard.
8. Wait until the instance is running and health checks settle.
9. Open the instance detail page and copy the Convex admin key.
10. Use the instance backend URL, dashboard URL, and admin key to connect the user's app or continue backend development.

After the admin key is available:
- The agent can help create schemas, functions, auth flows, app APIs, dashboards, seed data, and admin tools.
- The agent can help wire a mobile app to the new backend.
- The agent can help add Better Auth, push notification configuration, analytics, error tracking, backups, and database operations.
- Treat the admin key as a secret. Do not paste it into public chats, source code, or client-side mobile code.

Important operational cautions:
- Preserve Docker volumes. Do not run destructive volume removal unless the user explicitly wants to delete data.
- Convexer data lives in `/app/server/data` inside the container.
- The Docker volume must mount to `/app/server/data`, not `/app/server`.
- `instance.extra_env` is JSON stored in SQLite and must be parsed before use.
- Public API endpoints must be registered before authenticated `/api` routes.
- This setup uses non-TLS PostgreSQL inside the Docker network with `DO_NOT_REQUIRE_SSL=1`.
- Back up data before upgrades, restores, deletion, or risky database work.

Repository guide for implementation agents:
- Read `/README.md` first for install, architecture, DNS, backups, and env vars.
- Read `/AGENTS.md` for project-specific gotchas before editing code.
- Main dashboard frontend: `/client`
- Backend and Docker orchestration: `/server`
- Promo GitHub Pages site: `/promo`
- Installer: `/install.sh`
- Production Docker setup: `/Dockerfile` and `/docker-compose.yml`

Development commands:
- Root install: `npm install`
- Full dev: `npm run dev`
- Production build: `npm run build`
- Promo dev: `cd promo && pnpm install && pnpm dev`
- Promo build: `cd promo && pnpm build`

Agent behavior expectations:
- Be honest about tradeoffs: Convexer lowers backend platform cost but adds server ownership.
- Explain infrastructure decisions in plain language for non-technical users.
- Prefer small, reversible steps.
- Keep credentials private.
- Verify after each major setup step before moving on.
- If changing code, keep changes scoped and follow existing project conventions.
